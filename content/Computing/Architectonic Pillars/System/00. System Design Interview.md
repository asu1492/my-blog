1. Follow a framework: outline functional/non-functional requirements, core entities, APIs, high-level design, then deep dives for scalability and performance.
2. six core concepts: 
	1. storage (relational vs. document vs. key-value, ACID vs. BASE)
	2. scalability (vertical/horizontal, sharding, consistent hashing)
	3. networking (application/transport/network layers, REST/gRPC/WebSockets)
	4. latency/throughput (e.g., memory vs. disk) and performance 
	5. fault tolerance/redundancy
	6. CAP theorem (prioritize consistency or availability).
3. Core building blocks include
	1. servers
	2. databases (PostgreSQL, DynamoDB) 
	3. caches (Redis)
	4. message queues (Kafka)
	5. load balancers
	6. blob storage (S3)
	7. CDNs for global content delivery.
4. Practice these 10 in order: URL Shortener (Bitly), Dropbox, Ticketmaster, News Feed, WhatsApp, LeetCode, Uber, Web Crawler, Ad Click Aggregator, Facebook Post Search. Start passively for first few, then solve independently on a whiteboard before reviewing solutions; repeat for patterns.
5. Non Functional Needs 
	1. SARPSO-E
	2. Scalability
	3. Availability
	4. Reliability 
	5. Performance 
	6. Security
	7. Observability 
	8. Extensibility

1. Step 1: Understand the problem and Identify design scope (3 - 10 minutes) 
	1. Ask Questions
	2. Gather All Information 
	3. Make Assumptions and Write it down 
	4.  Overall goals and feature scope
	5. Clarify the scope by asking questions to narrow broad problems, outlining functional requirements (what's in/out of scope) and non-functional ones (availability, latency, security). Estimate data volumes like storage or user traffic, stating assumptions explicitly and confirming with the interviewer.
2. Step 2: HLD (10 - 15 minutes)
	1. Draw box diagram with key components APIs(contracts; which API's and why), how client and web server communicate, cache, data stores, etc. 
	2. Outline core components including APIs (REST, GraphQL, etc.) with request/response details, client-server communication, and a diagram showing data/control flow without early scalability focus.
	3. Including API endpoints and database schema depends on the problem. E.g. Google Search - LLD while multi player poker game - HLD 
	4. Start with RESTful APIs (or WebSockets for real-time needs), then draw a diagram including load balancers, services, and data storage without specifying exact databases yet. Develop data models considering access patterns and read/write ratios, review end-to-end feature coverage, and note discussion points for later
3. Step 3: Design deep dive (10 - 25 minutes)
	1. Examine components in detail, discussing how non-functional needs influence choices like partitioning databases or adding load balancers. Present multiple options with pros/cons, justifying preferences based on constraints
4. Step 4: Scaling & Bottlenecks
	1. Identify single points of failure, replication needs, global deployment, edge cases (e.g., peak usage), and scaling via sharding, CDNs, caching, or rate limiting to handle 10x growth
5. Step 5: Wrapup (3 - 5 minutes)
	1. Identify system bottlenecks and things for improvement. 
	2. Talk about error cases : server failure, network losses
	3. Operation issues : How to monitor metrics and error logs. 
	4. Summarize decisions, trade-offs (time/space), verify requirement coverage, and suggest improvements, leaving time for question
6. Things to keep in mind
	1. Make it more collaborative rather than didactic by following Socratic Method. 
	2. Avoid Over engineering 
	3. There are no right answer, approach is much more important. So, go slow and explain your design.
